# POI Marker Kit Project Plan

## 1. Objective 
Build a location-based-webxr kit that create beautiful markers given a real-world position:
- markers need to stay hidden when user is far away
- plays a striking reveal animation when the user gets close enough
- idles with subtle motion while in view
- grows into a focused state when the user taps or looks straight at it 
- plays a hide animation when the user walks away 
- visually positioned above the anchor point rather than exactly on it

## 2. Scope and Non-Goals

### In scope
- Marker lifecycle behavior
- Distance-based visibility
- Gaze and tap interaction
- Basic WebXR and non-WebXR rendering
- A demo/gallery page for local testing

### Out of scope
- Full geospatial mapping
- Multiplayer synchronization
- Advanced physics-based animations
- Full production analytics or persistence

## 3. Marker spatial behaviour 
- Markers must be offset vertically above their anchor point.
- Markers should not appear to be pinned exactly to the real-world coordinate; they should hover above the environment surface.
- Marker placement must be configurable by anchor height and offset. 

## 4. High-level Architecture
### 4.1 Application type: 
- Web page with a small fixed three.js scene
- Another scene, this time using location-based-webxr

### 4.2 Module structrue
- `src/main.ts`
    - App bootstrap
    - Initializes location based WebXR scene and renderer

- `src/config.ts`
  - Threshold values and timing constants
  - Example values:
    - `hideDistance`: 20m
    - `revealDistance`: 15m
    - `focusDistance`: 8m
    - `focusAngle`: 30 degrees
    - `focusDwellMs`: 500ms
    - `hoverHeight`: 2m

- `src/index.html`
    - Click through gallery
    - Fixed three.js test scene
    - Controllable camera
    - Loads markers 

- `src/markers/markerStateMachine.ts`
  - Owns per-marker lifecycle state transitions
  - Converts incoming spatial and interaction data into state changes

- `src/markers/markerController.ts`
  - Coordinates marker creation, update, and cleanup
  - Maintains a collection of marker runtime instances

- `src/interaction/attention.ts`
  - Detects whether a marker is in view
  - Computes whether it is the current gaze target

- `src/interaction/input.ts`
  - Handles click events
  - Converts input into marker focus requests

- `src/spatial/distance.ts`
    - Computes the distance between the user and the marker

- `src/spatial/POV.ts`
  - Reads and normalizes user POV data
  - Encapsulates position/orientation calculations

- `src/render/threeRenderer.ts`
  - Creates and updates Three.js objects
  - Applies state-based visual changes

- `src/scene/scene.ts`
  - Builds the scene graph
  - Separates WebXR renderer from the fallback renderer

## 5. State Design
### 5.1 States
- `hidden`
  - Marker is not visible
  - User is too far away or the marker is not yet eligible to render
- `revealing`
  - Marker transitions from hidden to visible
  - Animation plays once
- `idle`
  - Marker is visible
  - User is near enough
  - Marker is not currently focused
- `focused`
  - Marker is visible
  - User is near enough
  - Marker is in focus via gaze or tap
- `hiding`
  - Marker exits visible state
  - Animation plays once

### 5.2 State Transitions
- `hidden -> revealing`
  - Trigger: distance <= `revealDistance`
  - Guard: marker has been initialized and anchor data is valid

- `revealing -> idle`
  - Trigger: reveal animation completes

- `idle -> focused`
  - Trigger: marker is in view and either:
    - user taps the marker, or
    - gaze dwell time exceeds `focusDwellMs`

- `focused -> idle`
  - Trigger: focus is lost due to gaze shifting away, tap cancelled, or distance increased

- `idle -> hiding`
  - Trigger: distance > `hideDistance` or marker becomes invalid

- `focused -> hiding`
  - Trigger: distance > `hideDistance` or marker becomes invalid

- `hiding -> hidden`
  - Trigger: hide animation completes

### 5.3 Hysteresis 
- introduce hysteresis to avoid oscillation between states
  - use different threshold values between states


## 6. Redux state setup
### 6.1 Actions
- `COMPUTE_FOCUS`
    - Payload: `{boolean: inFocus}`
- `COMPUTE_DISTANCE`
    - Payload: `{int: distance}`
- `CHANGE_STATE`
    -Payload: `{MarkerState: newState}`

### 6.2 Reducer behaviour
- `COMPUTE_FOCUS`
    - Based on the current user's perspective, calculates if the object is in view.
- `COMPUTE_DISTANCE`
    - Based on the current position of the user and the marker, calculates the distance between the two.
- `CHANGE_STATE`
    - Trigger a transition between two states.

## 7. 3D Scene and camera designs 
### 7.1 Scene setup
- Create a Three.js `Scene` with a `PerspectiveCamera`.
- Add ambient + directional lighting for marker visibility.
- Create a WebXR `Scene`.
- Add `THREE.CanvasTexture` and render onto a plane for UI  
### 7.2 Camera controls
- Use Three.js `OrbitControls` from `three/examples/jsm/controls/OrbitControls.js`.
### 7.3 Rendering markers
- For each marker in the scene, create a THREE.js `Object3D` object.
- Use a fixed marker style for each state

## 8. Testing Strategy
### 8.1 Testing Principles

- Test behavior, not implementation details.
- Test logic separately from rendering.
- Use deterministic input data and fake timers where necessary.

### 8.2 Unit Tests
#### 8.2.1 Marker state machine
Must verify:
- correct state ordering for normal transitions
- correct transitions for reveal, idle, focus, and hide
- invalid transitions are rejected
- hysteresis prevents rapid oscillation
- timers are respected
- state changes are blocked when marker input is invalid or incomplete

#### 8.2.2 Distance logic
File: `src/spatial/distance.test.ts`

Must verify:
- correct Euclidean distance calculations
- threshold boundary behavior at exact reveal/hide/focus values
- invalid or missing position values are handled safely
- distance values are normalized correctly for comparison

#### 8.2.3 Attention and interaction logic
File: `src/interaction/attention.test.ts`

Must verify:
- marker is considered in view only when it is actually inside the view frustum
- gaze dwell only triggers once the dwell time threshold is exceeded
- transition to focus occurs when the conditions are properly met
- focus is cleared when the user looks away or moves out of range

#### 8.2.4 Pose and camera logic
File: `src/spatial/POV.test.ts`

Must verify:
- user position/orientation data is normalized correctly
- marker offset calculations are applied correctly

### 8.3 Rendering and Runtime Tests

Must verify:
- visual state matches runtime state
- marker objects are created and destroyed correctly
- reveal and hide animations are applied in the proper order
- marker placement uses the configured hover offset and anchor height
- object cleanup prevents orphaned scene nodes

### 8.4 Failure-Mode Tests
Must verify behavior for:
- WebXR unsupported
- camera permission denied
- location permission denied
- poor GPS accuracy
- invalid or missing marker data
- slow or failed asset loading
- interrupted animation due to scene reset or state change
- user leaving the interaction range during reveal or hide

### 8.5 Performance Tests
Must verify:
- marker updates remain stable for a reasonable marker count
- frame rate does not collapse under a basic test scene
- marker updates do not scale badly with the number of markers
- expensive visual updates are skipped or reduced when the device is under load

## 10. Behaviour handling
- GPS - AR misalignment
    - Prevent jumps, by correcting the position with a continious motion
        - avoid motions that are too slow to not create the feeling of delay.
        - avoid motion that are too fast to not create jumps
- Maintain readability
    - Find optimal values for thresholds, so that objects in focus are actually recognizable on the phone screen.
